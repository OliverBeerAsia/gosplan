# Art Authoring Template: Building Family Packs

This is the template for every future authored building-family art pack (panelak, kommunalka, stalinka, factory, and so on). It documents exactly how the khrushchyovka, the one fully-authored family, is built. When in doubt, read the real files:

- `public/assets/art/atlases/buildings-khrushchyovka-1x.svg` and `.json`
- `public/assets/art/atlases/environment-worker-housing-1x.svg` and `.json`
- the khrushchyovka building entry and three environment entries in `public/assets/art/manifest.v1.json`
- schema doc-comments in `src/graphics/ArtManifest.ts`
- style rules in `docs/ART_DIRECTION.md`

## Files in a family pack

One family ships four to six files plus manifest entries:

1. `public/assets/art/atlases/buildings-<family>-1x.svg` - the atlas image
2. `public/assets/art/atlases/buildings-<family>-1x.json` - the frames file
3. Optional: `environment-<theme>-1x.svg` and `.json` for courtyard compositions
4. Entries in `public/assets/art/manifest.v1.json`: one atlas entry per SVG, one building entry, optional environment entries
5. A per-family check script (`scripts/check-<family>-atlas.cjs`) modeled on `scripts/check-khrushchyovka-atlas.cjs`

## Building atlas layout

The khrushchyovka atlas is a fixed 480x528 SVG canvas holding a 3x3 grid of 160x176 cells:

- Rows are LOD levels top to bottom: `far` (y=0), `mid` (y=176), `near` (y=352)
- Columns are physical masses left to right: one column per variant (short slab, long slab, linked return)
- Every cell is untrimmed and transparent: the frame rect is always the full 160x176 cell, never shrink-wrapped to the drawing
- No terrain is baked into any cell; the building sits on transparent ground
- The root element declares `shape-rendering="crispEdges"` for pixel rendering
- Art is code-native SVG geometry only: no `<text>`, `<image>`, `<foreignObject>`, and no `url(...)` external paint (the khrushchyovka check enforces this)

The frames file maps `<mass>-<lod>` IDs onto that grid, colocated with the SVG:

```json
{
  "image": "buildings-khrushchyovka-1x.svg",
  "frames": {
    "short-far":   { "x": 0,   "y": 0,   "w": 160, "h": 176 },
    "long-far":    { "x": 160, "y": 0,   "w": 160, "h": 176 },
    "linked-far":  { "x": 320, "y": 0,   "w": 160, "h": 176 },
    "short-mid":   { "x": 0,   "y": 176, "w": 160, "h": 176 },
    "short-near":  { "x": 0,   "y": 352, "w": 160, "h": 176 }
  }
}
```

## Anchor coordinate space

All points in the manifest (anchor, entrances, queueAnchors, windowAnchors, emitter positions) are pixels inside the frame cell, origin at the cell's top-left, y-down. They are post-transform coordinates: measure against the rendered frame, not the pre-transform drawing coordinates inside your `<g transform=...>` groups.

Every building frame uses anchor `[80, 168]`: the horizontal center of the 160-wide cell, 8px above the cell bottom. That point is the ground-contact position the renderer pins to the building's tile. Keep it identical across all masses and LODs of a family so LOD swaps and variant selection never make the building slide.

## Manifest entries

Register each atlas, then the building. Real excerpts from the khrushchyovka entry:

```json
{
  "id": "buildings.khrushchyovka_1x",
  "image": "assets/art/atlases/buildings-khrushchyovka-1x.svg",
  "framesFile": "assets/art/atlases/buildings-khrushchyovka-1x.json",
  "pixelRatio": 1
}
```

Atlas IDs are stable and namespaced (`buildings.<family>_1x`, `environment.<theme>_1x`). Frame references everywhere else are `<atlasId>:<frameId>`. Do not reuse `legacy.pixel_city`; that ID is permanently excluded from runtime building selection.

```json
{
  "id": "building.khrushchyovka.authored.v1",
  "buildingId": "khrushchyovka",
  "footprint": [2, 1],
  "anchor": [80, 168],
  "iconFrame": "buildings.khrushchyovka_1x:short-near",
  "lod": {
    "far": "buildings.khrushchyovka_1x:short-far",
    "mid": "buildings.khrushchyovka_1x:short-mid",
    "near": "buildings.khrushchyovka_1x:short-near"
  },
  "variants": [ ... ],
  "proceduralFallback": "khrushchyovka"
}
```

Required metadata per new family:

- `buildingId` must exist in `src/buildings/BuildingRegistry.ts`, and `footprint` must equal that registry definition's width x height (the validator cross-checks both)
- `anchor` must lie inside the near frame bounds
- `iconFrame` is the frame the build toolbar shows; use the default mass at near LOD
- `lod` is the building-level default mass; each variant carries its own `lod` block
- `proceduralFallback` must be a key generated by `src/graphics/BuildingTextures.ts` (`textures.set('<key>', ...)`); it is the recovery path when the atlas fails to load
- Industrial families add `emitters`: `{ "kind": "smoke" | "steam" | "spark", "position": [x, y] }`, positions in the same frame-local space
- `constructionFrames`, `winterOverlay`, and `conditionOverlays` are optional and must match the near frame's dimensions exactly

Each variant is one physical mass:

```json
{
  "id": "linked_return",
  "frame": "buildings.khrushchyovka_1x:linked-near",
  "lod": {
    "far": "buildings.khrushchyovka_1x:linked-far",
    "mid": "buildings.khrushchyovka_1x:linked-mid",
    "near": "buildings.khrushchyovka_1x:linked-near"
  },
  "allowedDistricts": ["worker_housing", "heavy_industry", "scientific_city", "historic_core"],
  "minimumEra": 2,
  "weight": 1,
  "entrances": [[44, 137], [70, 145]],
  "queueAnchors": [[44, 142], [70, 150]],
  "windowAnchors": [[32, 101], [45, 101], [60, 101], [73, 101], ...]
}
```

- `frame` is the backward-compatible single-frame fallback and must be the variant's near LOD
- Anchors are owned by each variant, not the shared building definition (masses differ, so door and window positions differ); the khrushchyovka check fails if building-level `entrances`/`queueAnchors`/`windowAnchors` reappear
- Entrance points sit on visible door pixels, queue points immediately below the thresholds, window points inside the dark near-LOD panes
- The khrushchyovka authors 20 to 30 windowAnchors per mass as a regular grid (see `gridPoints` in `scripts/check-khrushchyovka-atlas.cjs`)

## District and era gating

`src/graphics/ArtVariantResolver.ts` selects variants deterministically:

- Eligibility filter: a variant is eligible when the building's district style is in `allowedDistricts`, `era >= minimumEra`, and `era <= maximumEra` when present. A building outside any district (`'none'`) gets no authored variant and falls back to procedural art.
- Selection: an FNV-1a hash over `mapSeed`, building `id`, `defId`, `gx`, `gy` picks from the eligible set by `weight`, after sorting by variant ID so manifest reordering changes nothing.
- District style is deliberately excluded from the hash: rezoning must not morph an existing building's physical mass. Era and district only filter eligibility.
- No `Math.random` and no simulation RNG (the validator scans the resolver source for both). Same seed and placement always yields the same mass, on every machine and after every load.

LOD is derived from zoom (`far` below 0.45, `near` from 0.9, `mid` between) with 0.05 hysteresis, and is never stored in save data.

## Environment compositions

Environment defs are cosmetic, building-owned courtyard recipes. They never enter Grid or save data; owner identity plus map seed recreates them after load. The worker-housing environment atlas uses 64x80 cells with anchor `[32, 72]`, columns base far/mid/near then winter far/mid/near, one row per prop type.

The manifest entry (`environment.worker_housing.service_edge`, abridged):

```json
{
  "id": "environment.worker_housing.service_edge",
  "ownerBuildingIds": ["khrushchyovka", "panelak", "kommunalka", "stalinka"],
  "footprint": [2, 1],
  "placements": [
    {
      "id": "north_east_road_edge",
      "offset": [2, 0],
      "requiresRoadWithin": 1,
      "roadEdge": "north_east",
      "variantIds": ["north_east_service"]
    }
  ],
  "variants": [
    {
      "id": "north_east_service",
      "weight": 1,
      "parts": [
        {
          "id": "path_shelter_ne",
          "offset": [0, 0],
          "layer": "ground_decal",
          "anchor": [32, 72],
          "lod": { "far": "environment.worker_housing_1x:walkway-base-far", "mid": "...", "near": "..." },
          "winterLod": { "far": "environment.worker_housing_1x:walkway-winter-far", "mid": "...", "near": "..." },
          "proceduralFallback": "prop_none",
          "minimumQuality": "high"
        }
      ]
    }
  ],
  "powerSource": "none"
}
```

Semantics:

- `placements` are candidate positions; `offset` is grid-space from the owning building's master cell (negative values allowed). `requiresRoadWithin` is a same-elevation Manhattan road search radius. `roadEdge` pins orientation exactly: `north_east` requires a road along gy - 1, `north_west` along gx - 1. `variantIds` restricts a placement to orientation-safe variants.
- `variants` are weighted recipes; `parts` are tile-local pieces. Part `offset` is grid-space inside the definition's `footprint` and must stay inside it.
- `layer` is `ground_decal` (renders under props, walkways and pads) or `prop` (depth-sorted with buildings, benches and kiosks).
- `winterLod` swaps frames in winter; other seasons use base `lod`. `emissiveLod` is a transparent night-light overlay gated by power when `powerSource` is `"owner"` (`"none"` means no power gate).
- `minimumQuality` (`medium` or `high`) drops decorative parts on lower graphics tiers; ground paths use `high`, primary props `medium`.
- Every part declares a `proceduralFallback` TextureFactory key (`prop_none` is valid for pure decals).

## Validation gates

All of these must pass before a pack merges:

- `npm run check:art` - full manifest validation: frame references resolve, frames stay inside image bounds, footprints match BuildingRegistry, anchors stay inside frames, district/era/weight sanity, procedural fallbacks exist, and every SVG atlas or UI/loading asset is well-formed XML (duplicate attributes, mismatched or unclosed tags, and unquoted attribute values are hard errors)
- `npm run test:art` - node:test suite over the validator and resolver, including determinism of variant hashing
- A per-family check, e.g. `npm run check:khrushchyovka` - locks the reviewed contract: canvas size, crispEdges, no text/image/external paint, exact frame grid, fixed anchor, per-variant anchor tables. Copy `scripts/check-khrushchyovka-atlas.cjs`, replace the constants and anchor contract, and register a `check:<family>` script in `package.json`
- `npm run check:visual-evidence` - verifies the captured PNG evidence in `docs/graphics-implementation-evidence/`

## New family checklist

1. Author the SVG atlas: 160x176 untrimmed transparent cells, rows far/mid/near, one column per mass, anchor [80, 168], no baked terrain, crispEdges, geometry only.
2. Write the frames JSON: `<mass>-<lod>` IDs, full-cell rects, `image` pointing at the colocated SVG.
3. Add the manifest atlas entry (`buildings.<family>_1x`, pixelRatio 1) and the building entry: footprint and buildingId matching BuildingRegistry, iconFrame, base lod, per-variant lod plus districts, eras, weight, entrances, queueAnchors, windowAnchors, proceduralFallback, and emitters for industrial families.
4. Optional: author the environment atlas (64x80 cells, anchor [32, 72], base plus winter columns) and environment defs with placements, variants, and parts.
5. Run the validators: `npm run check:art`, `npm run test:art`, and your new `check:<family>` script.
6. Capture visual evidence at 0.25x, mid, and 1x zoom across seasons into `docs/graphics-implementation-evidence/` and verify with `npm run check:visual-evidence`.
